\section{Propuesta de simulación}

En esta sección, previa a la implementación, se describe el escenario diseñado para el desarrollo del trabajo.

El objetivo principal de este entorno es servir como material práctico de entrenamiento para analistas de seguridad, permitiendo la reconstrucción de un incidente realista a partir de un conjunto controlado de artefactos. Para ello, se define la infraestructura de la organización víctima y se pone a disposición el conjunto completo de evidencias generadas durante la simulación del incidente.

El propósito del analista es reconstruir la cadena completa del ataque mediante el análisis de dichos artefactos, identificando las técnicas empleadas, correlacionando la información disponible y comprendiendo la evolución del compromiso desde el acceso inicial hasta el establecimiento de la persistencia y el control del sistema.

Desde una perspectiva general, la simulación se ha diseñado como un escenario de alta complejidad técnica, con el objetivo de reproducir un entorno lo más cercano posible a un incidente real. Esto permite que el trabajo no se limite a un único vector de ataque, sino que integre múltiples fases del ciclo de intrusión, incrementando el valor formativo del ejercicio y la dificultad del análisis forense.

\section{Descripción del Atacante}

En este trabajo no se modela un adversario real ni se replica el comportamiento de un \acrshort{apt} existente. En su lugar, se diseña un atacante ad hoc, construido específicamente para este escenario con fines didácticos. Este enfoque permite ajustar de forma controlada sus tácticas, técnicas y decisiones operativas para alinearlas con los objetivos formativos del ejercicio.

    \label{fig:ataque}

El escenario propuesto reproduce un ataque estructurado en dos fases, diseñado para simular el comportamiento de un adversario realista.

En la primera fase, el atacante obtiene acceso inicial al sistema víctima mediante el uso de un dispositivo físico que le permite ejecutar código de forma inmediata. A partir de este punto de entrada, establece una conexión remota hacia su infraestructura externa, desde la cual realiza un reconocimiento básico del sistema y prepara la siguiente etapa del ataque.

En la segunda fase, el adversario despliega su componente principal, diseñado para operar de forma persistente y con baja visibilidad dentro del sistema. Este componente establece mecanismos de persistencia, amplía las capacidades de control remoto y se integra en procesos legítimos del sistema con el objetivo de dificultar su detección. A partir de este momento, el atacante puede realizar acciones más avanzadas, como el análisis del entorno, la escalada de privilegios, el movimiento lateral o la exfiltración de información.

\section{Descripción técnica de la simulación}

A nivel técnico, el incidente simulado se estructura en dos fases diferenciadas que reflejan la progresión del compromiso y la complejidad creciente del ataque.

\textbf{Fase 1: acceso inicial y establecimiento de acceso remoto.}  
La primera fase comprende la intrusión inicial en el sistema. El acceso se realiza mediante un dispositivo \textit{BadUSB}, seguido de la evasión de mecanismos de protección mediante un \textit{\acrshort{amsi} Bypass} basado en \textit{Memory Patching via Reflection}. Como resultado de esta fase, se consigue establecer una \textit{Reverse Shell} implementada en ensamblador (\textit{ASM}), que proporciona control remoto básico sobre el sistema comprometido.

\textbf{Fase 2: ejecución avanzada, evasión y persistencia en memoria.}  
A partir de la \textit{Reverse Shell} obtenida en la fase anterior, se desarrolla la segunda fase del ataque, centrada en técnicas avanzadas de ejecución en memoria, evasión y persistencia. En esta etapa se emplea \textit{\acrshort{dll} Hijacking} como técnica principal de weaponización, combinando \textit{Side Loading}, \textit{\acrshort{dll} Proxying} y \textit{Reflective Loading}. Estas técnicas permiten la ejecución de código dentro de procesos legítimos, minimizando la huella en disco y favoreciendo un enfoque \textit{fileless}.

Asimismo, se utiliza \textit{Reflective Loading} para la carga del módulo de \textit{Sliver \acrshort{c2}} directamente en memoria, evitando el uso de \texttt{LoadLibrary} y eliminando la necesidad de escribir artefactos en disco. De este modo, el canal de mando y control se establece completamente en memoria, incrementando significativamente la dificultad de detección y análisis forense.

Finalmente, se incorporan mecanismos de persistencia basados en \textit{Living-off-the-Land Binaries} (LOLBins), aprovechando herramientas legítimas del sistema para mantener el acceso en el tiempo sin introducir binarios adicionales en el sistema comprometido.

La combinación de ambas fases permite modelar un escenario realista de intrusión avanzada, en el que se evoluciona desde un acceso inicial clásico hasta una operación completamente orientada a la ejecución en memoria, la evasión de controles de seguridad y el control persistente del sistema.

Para estructurar este proceso, el trabajo se organiza en dos subsecciones complementarias:

\subsection{El Incidente \ref{chap:incidente}}
En este capítulo se describe de forma secuencial la actividad del atacante, incluyendo sus objetivos, las técnicas MITRE ATT\&CK asociadas y las evidencias donde pueden observarse.

\subsubsection{Indice de fases del ataque}

\begin{itemize}
    \item \textbf{Reconocimiento (Reconnaissance) \ref{sec:reconocimiento}:} 
    Recopilación de información sobre el entorno objetivo mediante análisis de superficie expuesta, infraestructura Active Directory en GOAD-Light y servicios accesibles de la workstation.
     
    \item \textbf{Preparación de armas (Weaponization) \ref{sec:armamento}:} 
    Desarrollo del payload en PowerShell, construcción del loader en .NET y preparación del implante de Sliver junto con la infraestructura de \acrshort{c2}.
 
    \item \textbf{Entrega (Delivery) \ref{sec:entrega}:} 
    Introducción del payload inicial mediante dispositivo BadUSB emulando teclado \acrshort{hid}, ejecutando comandos automáticamente en la máquina víctima sin interacción del usuario.

    \item \textbf{Explotación (Exploitation) \ref{sec:explotacion}:} 
    Ejecución inicial de código mediante script de PowerShell que descarga y lanza un payload en \texttt{\$env:TEMP}, estableciendo una reverse shell hacia el servidor del atacante.
 
    \item \textbf{Instalación (Installation) \ref{sec:instalacion}:} 
    Despliegue de \acrshort{dll} maliciosa mediante \acrshort{dll} hijacking en el binario de Calibre y establecimiento de persistencia mediante ejecución del payload en cada inicio del sistema.

    \item \textbf{Comando y Control (Command and Control, \acrshort{c2}) \ref{sec:comando-control}:} 
    Comunicación persistente del implante de Sliver con infraestructura externa mediante beaconing HTTPS cifrado, permitiendo ejecución remota de comandos y exfiltración de resultados.

    \item \textbf{Acciones sobre el objetivo (Actions on Objectives) \ref{sec:acciones-sobre-objetivo}:} 
    Ejecución del componente malicioso desde el implante en memoria (.NET), realizando una acción final controlada consistente en la creación de un fichero en el escritorio del usuario como indicador de compromiso, simulando el impacto de una fase de ransomware sin ejecutar operaciones destructivas.
\end{itemize}

No se trata de una investigación completa, sino de una referencia estructurada que ayuda a entender cómo se manifiestan las técnicas del atacante en el sistema comprometido.

\subsection{Red Team Notes \ref{chap:red-team}}
Este capítulo ofrece una visión complementaria desde la perspectiva del atacante, describiendo el desarrollo técnico de las herramientas, técnicas y procedimientos empleados durante la simulación del ataque. Se profundiza en las decisiones de diseño, las implementaciones realizadas y los mecanismos utilizados para maximizar el sigilo y la eficacia de la intrusión.

\subsubsection{Índice de contenidos}

\begin{itemize}

    \item \textbf{Acceso Inicial: Justificación del uso de BadUSB \ref{sec:bad-usb}:}
    Explicación de la elección de un dispositivo BadUSB como vector de acceso inicial, así como las ventajas operativas de la emulación de un teclado \acrshort{hid} y las técnicas utilizadas para evadir mecanismos de protección como \acrshort{amsi}.

    \item \textbf{Programación de shellcodes \ref{sec:shellcodes}:}
    Desarrollo de shellcodes independientes de posición (\acrshort{pic}), técnicas de runtime linking, diferencias entre las \acrshort{abi} de arquitecturas x86 y x64 y construcción de los payloads finales utilizados durante la operación.

    \item \textbf{Inyección de código \ref{sec:inyeccion}:}
    Estudio de las técnicas empleadas para ejecutar código dentro de procesos legítimos, centrándose en el uso de \acrshort{dll} Hijacking como mecanismo de ejecución.

    \item \textbf{Weaponización de \acrshort{dll} Hijacking \ref{sec:weaponizacion-dll}:}
    Análisis de distintas estrategias para convertir el \acrshort{dll} Hijacking en un mecanismo de ejecución robusto, incluyendo Side Loading, \acrshort{dll} Proxying y Reflective Loading.

    \item \textbf{Syscalls y evasión de \acrshort{edr} \ref{sec:syscalls}:}
    Comparativa entre los distintos niveles de acceso al sistema (Win32 API, Native API, Direct Syscalls e Indirect Syscalls), analizando su impacto en la evasión de soluciones \acrshort{edr} y mecanismos de monitorización.

    \item \textbf{Persistencia \ref{sec:persistencia}:}
    Descripción de los mecanismos implementados para mantener el acceso al sistema comprometido tras la ejecución inicial del implante.

    \item \textbf{Infraestructura \acrshort{c2} \ref{sec:c2}:}
    Configuración y despliegue de la infraestructura de Command and Control basada en Sliver, incluyendo la generación de implantes, listeners y comunicaciones cifradas.

    \item \textbf{Acciones finales \ref{sec:acciones-finales}:}
    Presentación de las técnicas utilizadas para la ejecución de capacidades avanzadas desde el implante, incluyendo el uso de \acrshort{bof}/\acrshort{coff}, Shared Library Side Loading y assemblies .NET como mecanismos de post-explotación.

\end{itemize}

Este capítulo no pretende describir cronológicamente la intrusión, sino documentar las técnicas y desarrollos implementados desde la perspectiva del Red Team, proporcionando el contexto técnico necesario para comprender el funcionamiento interno de cada componente utilizado durante la simulación.

\subsection*{Infraestructura de la víctima}

Como base del entorno corporativo se utiliza \textbf{GOAD-Light}, una versión reducida del laboratorio GOAD orientada a entornos de Active Directory vulnerables. Este entorno proporciona un dominio Windows preconfigurado, servicios esenciales y múltiples debilidades intencionadas que permiten simular técnicas reales de compromiso y movimiento lateral.

Las características completas del entorno GOAD-Light están documentadas por sus autores \cite{goad-light}.

Dentro del dominio se incluye una \textbf{estación de trabajo Windows 11}, completamente actualizada y con las protecciones de seguridad nativas habilitadas. Esta máquina actúa como punto de entrada inicial del ataque y como principal superficie de interacción con el usuario víctima.

Con el objetivo de aumentar la visibilidad del análisis forense y enriquecer el conjunto de artefactos disponibles, se incorporan dos componentes adicionales:

\begin{itemize}
    \item \textbf{Suricata}: desplegado para la monitorización del tráfico de red, permitiendo detectar y correlacionar conexiones sospechosas, tráfico de comando y control (\textit{\acrshort{c2}}) y patrones de actividad maliciosa.
    
    \item \textbf{Sysmon}: instalado en la estación de trabajo para generar telemetría avanzada de eventos en Windows, incluyendo creación de procesos, carga de \acrshort{dll}s, conexiones de red y modificaciones en el registro, facilitando la reconstrucción temporal del ataque.
\end{itemize}

\subsection{Cómo utilizar este recurso}

Este recurso está pensado para servir como guía de apoyo durante el análisis del incidente y como material de referencia técnica para comprender en detalle cómo se desarrolló el ataque.

El contenido se organiza en dos bloques complementarios:

\begin{itemize}
    \item \textbf{Capítulo \ref{chap:background}: Background.}  
    Vocabulario técnico e instituciones sobre las que se apoya el contenido del trabajo.

    \item \textbf{Capítulo \ref{chap:incidente}: Reconstrucción del ciberataque.}  
    Presenta una visión estructurada del ataque, mapeando cada acción del adversario con las técnicas MITRE ATT\&CK aplicadas y con los artefactos del sistema víctima donde podría encontrarse evidencia relevante. Este bloque está orientado a facilitar el trabajo del analista \acrshort{dfir} durante el proceso de correlación y reconstrucción de los hechos.

    \item \textbf{Capítulo \ref{chap:red-team}: Red Team Notes.}  
    Describe el ataque desde el punto de vista del adversario, explicando el razonamiento, las decisiones técnicas y los procedimientos utilizados en cada fase. Ofrece una visión interna del flujo del ataque y del funcionamiento de cada componente ofensivo.
\end{itemize}

El proyecto está diseñado para que el analista pueda avanzar de forma autónoma utilizando únicamente las evidencias del caso. Si en algún momento necesita orientación sobre dónde buscar rastros de una técnica, puede consultar el Capítulo \ref{chap:incidente}. Si, en cambio, desea entender en profundidad por qué el atacante actuó de una determinada manera o cómo funciona una técnica concreta, puede acudir al Capítulo \ref{chap:red-team}, donde se detalla su implementación desde la perspectiva del red team.
